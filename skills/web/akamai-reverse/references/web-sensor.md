# Web Akamai sensor_data Methodology

> An honest preface: **there is no universal generator for the Web side.** The Akamai collection script is customized per site, changed frequently, the canvas cannot be faked, and it relies heavily on real behavior, on top of TLS fingerprinting + IP reputation. This is px-reverse-style **per-target hard work**.
> This document provides methodology + priorities, not "copy-paste and pass" code. Based on xiaoweigege/akamai2.0-sensor_data's analysis of maersk.com + the public Akamai _abck handshake documentation.

## 0. Priority: first ask "do I need pure algorithmic reverse engineering?"

Ordered from lowest to highest cost; if an earlier option solves it, don't move on:

1. **Real browser automation** (`cdp-browser`) — real Chrome with no webdriver traces + a real fingerprint + real behavior + a residential IP. Akamai is extremely lenient toward real browsers. **80% of Web scenarios are covered by this step.**
2. **Run the sensor in a browser + export the cookie to the backend** (semi-automatic) — the browser is responsible only for passing Akamai and obtaining a `~0~`-state `_abck`, while business requests are handled by the backend at high concurrency.
3. **node_bridge runs the obfuscated script** (`node-bridge-build`) — run the collection script in node + a stubbed environment to produce real sensor_data without restoring the algorithm. Less effort than pure AST restoration.
4. **Pure algorithmic AST restoration** — the most expensive. AST-deobfuscate the obfuscated script → restore each field of the 58-element array → a real canvas pool + behavioral simulation. Only worth it when you need extremely high concurrency / a browserless environment.

## 1. Cookie roles

| cookie | Role | When it appears |
|---|---|---|
| `_abck` | core verification state, the return value of sensor_data | issued on the first page load, updated after the sensor POST |
| `bm_sz` | Bot Manager session, one of the inputs to sensor generation | issued by the edge on the first page load |
| `ak_bmsc` | edge session state | first page load |
| `bm_sv`/`bm_mi`/`bm_lso` | auxiliary session | depends on the site |
| `sbsd`/`sbsd_o` | newer secondary enhanced verification | Akamai v3+ sites |

## 2. The _abck state machine (detailed)

```
GET homepage → Set-Cookie: _abck=<token>~-1~-1~...      ← invalid state
POST sensor_data (with the latest _abck + bm_sz)
  ├─ _abck contains ~0~        → valid, stop, request the business endpoint
  ├─ _abck ends in ~-1~-1      → keep POSTing
  └─ site does not use ~0~     → considered ready after a fixed 3 POSTs
Business endpoint 403 + re-issued ~-1~-1 → that _abck is invalidated, redo the handshake (usually a single sensor recovers it)
```

Implementation points:
- Use a persistent session (curl_cffi `Session` + impersonate) to keep cookies written back automatically.
- Before each POST, confirm it carries the **latest** `_abck`/`bm_sz`.
- Monitor response latency: when the sensor is not realistic enough, Akamai "lets it through but slows it down" (on the order of 10s), and a 200 status code does not mean high trust.

## 3. sensor_data structure (from the maersk.com analysis)

- Generated by concatenating a **58-element array** and then encrypting it.
- The two most critical items: the **canvas fingerprint** + the **mouse movement trajectory**.
- **The canvas fingerprint cannot be randomly faked** → collect a real browser's canvas value to substitute in.
- Movement trajectory → algorithmically simulate real mouse movement/clicks.
- `akamai-bm-telemetry` (request header) = a base64 variant of sensor_data, generated simply (base64 of sensor_data).

## 4. The AST restoration route (the most expensive one, only when necessary)

1. Use CDP to capture the sensor POST + download the obfuscated collection script, **recording the sha256 and pinning the version**.
2. AST deobfuscation (acorn / babel) — restore the obfuscated script into readable JS (refer to the ruishu-reverse AST methodology: build a function mapping table + recursively trace the call chain).
3. Locate the assembly point of the 58-element array and annotate each field: fixed / canvas / trajectory / time / random.
4. Data-driven: collect several batches of real sensors from the same version and compare field by field to find the source of each segment (do not try to hard-read the inner VM).
5. Connect a real canvas pool + behavioral simulation and pass the handshake state machine.

## 5. The sbsd secondary handshake (new sites)

- After upgrading to v3, Akamai introduced `sbsd`/`sbsd_o`, an enhanced verification type independent of sensor_data.
- First confirm whether the target business endpoint **actually requires** sbsd (many sites do not enable it).
- If enabled, you must reverse its payload separately — refer to the sbsd support notes in xiaoweigege/akamai2.0-sensor_data. Treat it as a second sensor type; do not mix it into the sensor_data logic.

## 6. An honest difficulty assessment

| Factor | Reality |
|---|---|
| Script universality | ❌ customized per site, none universal |
| Version stability | ❌ frequent version changes, old restorations break |
| canvas | ❌ cannot be faked, requires a real pool |
| behavior | requires real / lifelike trajectories |
| TLS | must use curl_cffi/tls-client/browser |
| IP | residential IP, rotate IPs |
| Conclusion | **prefer browser automation; pure algorithmic reverse engineering is a per-target long-term investment** |

## 7. Companion skills

- `cdp-browser`: capture the sensor POST + browser automation (first choice)
- `node-bridge-build`: run the obfuscated script in a node stubbed environment to produce the sensor (second choice)
- `web-api-analyzer`: locate the sensor submission endpoint and the business endpoint
