# Master Playbook: End-to-End (CDP Capture → Generator → Test)

> You've been handed a **brand-new PX-protected site** (or an existing site whose SDK
> has updated), and you need the **complete flow from zero to a working 10/10 cookie**.
>
> This playbook is the **integration guide** that ties the other 7 playbooks together,
> sequencing them in chronological order.
>
> Estimated total time: **6-12 hours** (6h when smooth, 12h when you hit snags).

## Stage 0: Launch Chrome via cdp skill, Capture Initial Traffic (15 min)

**Goal**: Determine whether the target site is PX-protected, and capture one collector
POST as a sample.

```bash
# 1. Launch CDP Chrome (real browser, no webdriver fingerprint)
python ../cdp-browser/scripts/cdp.py start

# 2. Navigate
python ../cdp-browser/scripts/cdp.py navigate "https://www.<target-site>.com"

# 3. Capture 20 seconds of all network traffic
python ../cdp-browser/scripts/cdp.py network 20 | grep -iE 'px|sensor\.|collector' > raw_traffic.txt
cat raw_traffic.txt | head -20
```

**Determine whether this site uses PX**:

```bash
# Check raw_traffic.txt for these URL patterns
grep -E "px-cloud\.net|/api/v2/collector|/xhr/api/v2/collector|sensor\.[a-z]+\.com" raw_traffic.txt
```

- ✅ Match → PX-protected, continue
- ❌ No match → not PX; this playbook does not apply

### Failure Fallback

| Symptom | Fix |
|---|---|
| Chrome won't launch | Check whether the cdp port is occupied (9222/9223) — `pkill chrome` and restart |
| `cdp.py navigate` returns 403 immediately | TLS fingerprint flagged by the CDN → check that Chrome is not in headless mode |
| No px-cloud requests visible but the page loads fine | Wait longer (the PX SDK may load lazily), or PX uses a first-party deployment (search `sensor.<host>.com`) |

## Stage 1: Download SDK + Identify Version (10 min)

See [`identify-sdk-version.md`](identify-sdk-version.md) for the full flow.

```bash
# 1. Extract the SDK URL from Stage 0's raw_traffic
SDK_URL=$(grep -oE 'https?://[^"]*\.(min\.)?js[^"]*' raw_traffic.txt | head -1)
echo "SDK URL: $SDK_URL"

# 2. Download the SDK
curl -sS "$SDK_URL" > sdk.js
sha256sum sdk.js
# → lock this version's SHA

# 3. Run a 5-point probe to confirm this is PX
for c in 1732584193 909522486 122192928e5 2147483647 F@bt; do
    echo "$c: $(grep -c "$c" sdk.js)"
done
# 5/5 hits = standard PX SDK
```

At the same time, extract the 5 constants from Stage 0's captured collector POST body
(see [`extract-constants.md`](extract-constants.md)):

```bash
# Assumes raw_traffic.txt contains a full collector POST body
APP_ID=$(grep -oE 'appId=[^&]+' raw_traffic.txt | head -1 | cut -d= -f2)
TAG=$(grep -oE 'tag=[^&]+' raw_traffic.txt | head -1 | cut -d= -f2 \
    | python -c 'import sys,urllib.parse;print(urllib.parse.unquote(sys.stdin.read()))')
FT=$(grep -oE 'ft=[^&]+' raw_traffic.txt | head -1 | cut -d= -f2)
echo "APP_ID=$APP_ID TAG=$TAG FT=$FT"
```

## Stage 2: Pin SDK Version (5 min)

**Goal**: Ensure the 6 capture batches below **all use the same SDK**. Without pinning,
each batch's hQ dictionary content differs, and the diff comes out as pure noise.

```bash
# Store the SDK into the project
SITE="<site>"   # replace with the site name (e.g. ifood)
mkdir -p "source"
cp sdk.js "source/main.min.js"

# Compute SHA + write SDK_INFO.md (see source/SDK_INFO.md format)
SHA=$(sha256sum "source/main.min.js" | cut -d' ' -f1)

cat > "source/SDK_INFO.md" <<EOF
# $SITE — Locked SDK Snapshot

| Property | Value |
|---|---|
| File | \`main.min.js\` |
| **SHA-256** | \`$SHA\` |
| Size | $(wc -c < source/main.min.js) bytes |
| Source URL | $SDK_URL |
| Captured | $(date -u +%Y-%m-%dT%H:%M:%SZ) |

## Constants
| Constant | Value |
|---|---|
| AppID | \`$APP_ID\` |
| TAG | \`$TAG\` |
| FT | \`$FT\` |
| Cookie | \`_px2 or _px3\` |
EOF
```

### Chrome Local Overrides (Recommended)

DevTools → Sources → Overrides → point at `source/`. From then on, every
Chrome request for `client.px-cloud.net/.../main.min.js` returns **this local file**.
**This locks all 6 capture batches to the exact same SDK.**

## Stage 3: Capture 6 Batches of Cold-Visit Samples (15-30 min)

```bash
# Reuse the capture script in ../cdp-browser/ (or adapt a new platform version)
cp ../cdp-browser/scripts/capture_via_cdp_ifood.py \
   ../cdp-browser/scripts/capture_via_cdp_<site>.py
# Edit the URL / cookie name / output path inside

# Capture 6 batches
for i in 1 2 3 4 5 6; do
    python ../cdp-browser/scripts/capture_via_cdp_<site>.py $i
    sleep 15   # avoid IP throttle
done
```

Each batch outputs to `samples/N/`:

```
samples/1/
├── request_1.txt   POST #1
├── request_2.txt   POST #2
├── request_3.txt   POST #3
├── response_1.json
├── response_2.json
├── response_3.json
└── meta.json   (contains sdk_sha256)
```

### 6-Batch SDK Consistency Check

```bash
for i in 1 2 3 4 5 6; do
    jq -r '.sdk_sha256' samples/$i/meta.json
done | sort -u
# Must print exactly one line; otherwise Stage 2's SDK pinning didn't take effect
```

## Stage 3.5: ⭐ Deployment-Tier Triage (5 min, NEW 2026-05-25)

> Historically the skill assumed all PX deployments were equally lenient (based on
> iFood/Grub experience), which caused strict-tier deployments (such as totalwine) to
> reach the end of every stage with a generator that still couldn't be used for real.
> This step **triages early** and sets the later validation strength. For the full
> comparison table see [`../references/deployment-tiers.md`](../references/deployment-tiers.md).

```bash
# 1. POST count (lenient = 2; strict = 3+)
jq '.collector_post_count' samples/*/meta.json

# 2. Collector path (first-party = high probability of strict tier)
jq -r '.first_collector_url' samples/1/meta.json

# 3. Check whether the 3rd POST body contains a cookie-echo field
if [ -f samples/1/decoded_payload_3.json ]; then
    jq '.[0].d | keys[]' samples/1/decoded_payload_3.json | \
        grep -E "OkpJAH8oTTA=|.*(eyJ1IjoiZ)" && \
        echo "⚠️  strict signal: EV3 contains cookie echo → must implement EV3 / seq=2 POST"
fi

# 4. Does state contain hid (from the OlllOOll segment)
grep -lE "OlllOOll" samples/*/decoded_response_1.json | wc -l
# 6/6 → state has hid; Gotcha #16 applies
```

**Write the triage result to the top of your project's `log/RECON.md`**:

```markdown
## Deployment Tier (Stage 3.5, YYYY-MM-DD)
- POST count: <2 | 3+>
- Collector path: <third-party px-cloud.net | first-party>
- EV3 cookie echo: <yes | no>
- state.hid: <yes | no>
- **Tier**: <lenient | strict>
- **Additional gotchas to watch**: <which of #15-18 apply>
```

**If strict tier**: during Stage 5-8, additionally watch for:
- Gotcha #16: implement EV3 (seq=2) + extract state.hid
- Gotcha #17: dictionary subfield (counter) synchronization
- Gotcha #18: all HMAC fields go through [`recover-hmac-formulas.md`](recover-hmac-formulas.md), crypto-verified across 6 batches
- Stage 8 must include Layer 3.5 (use the cookie to actually hit a PX-gated endpoint)

## Stage 5: Decoder — Directly Reuse the AI_re/reverse Modules (30 min)

**Core strategy**: PX algorithms haven't changed in 3 years, so **directly reuse the
9 modules in `revers/`**.

```bash
# 1. Decode all payloads + responses across all 6 batches
for i in 1 2 3 4 5 6; do
    node ../scripts/decode_payload.js  samples/$i/request_1.txt \
        > samples/$i/decoded_payload_1.json
    node ../scripts/decode_payload.js  samples/$i/request_2.txt \
        > samples/$i/decoded_payload_2.json
    node ../scripts/decode_response.js "$TAG" samples/$i/response_1.json \
        > samples/$i/decoded_response_1.json
    node ../scripts/decode_response.js "$TAG" samples/$i/response_2.json \
        > samples/$i/decoded_response_2.json
done

# 2. Run closed-loop validation (confirm the decoder is generic for the current SDK)
for i in 1 2 3 4 5 6; do
    node ../scripts/verify_batch.js samples/$i
done
# All PASS → decoder is 100% generic, proceed to Stage 6
```

### Decode Failure → The Algorithm Layer Has a Problem, Go to reverse-algorithms.md

If `decode_payload` or `decode_response` fails:

| Decode failure location | Which section to read |
|---|---|
| payload base64 → garbage at the XOR stage | [`reverse-algorithms.md`](reverse-algorithms.md) §1-2-3 (MD5/HMAC/XOR) |
| OB segment decodes to garbage | [`reverse-algorithms.md`](reverse-algorithms.md) §6 (ml() / OB XOR key) |
| Field names are wrong | [`reverse-algorithms.md`](reverse-algorithms.md) §5 (base91 hQ dictionary) |

**Rare case**: PX changed an algorithm (not seen in 3 years) — you'd have to fully
re-reverse the algorithms.

```bash

In the new generator, change 5 things:

2. Template (use batch 1 as the STATIC template)
3. state.* → EV2 b64 key mapping (use the output from Stage 6.2)
4. DYNAMIC field names (use the output from Stage 6.3)

d['<state.no key>'] = parseInt(ctx.state.no);   // number ⭐⭐⭐
```

## 🚨 Failure Fallback: Pure Static SDK Reversing (Reverse Independently from Experience)

If at any point in the flow above you **fail to capture, the decoder doesn't work, or you
can't locate fields**, fall back to **pure static SDK analysis**:

### Failure Scenario 1: Can't Capture the collector POST (Stage 0 / 3 failed)

Possible causes + fixes:

```
TLS fingerprint blocked by the front layer
  → use real Chrome (not headless) + CDP, don't send directly from a script
  → or the target site uses a Cloudflare front layer — you need to solve Cloudflare first

PX uses an unusual endpoint
  → search for fetch / XMLHttpRequest calls in the SDK
  → grep -nE 'fetch\(|new XMLHttpRequest' main.min.js
  → check whether the actual POST URL is a standard collector

No PX SDK loads at all
  → the target site doesn't use PX (check other fingerprints: Akamai/Cloudflare/DataDome)
```

### Failure Scenario 2: Decoder Doesn't Work on the New SDK

Probe the 5 magic constants to check whether it's PX:

```bash
for c in 1732584193 909522486 122192928e5 2147483647 F@bt; do
    echo "$c: $(grep -c "$c" main.min.js)"
done
```

- 5/5 → it's PX, the algorithm layer should be generic → check the OB XOR key (ml() input may not be TAG)
- < 5 hits → major PX revision (extremely rare) → work through the full [`reverse-algorithms.md`](reverse-algorithms.md)

### Failure Scenario 3: Can't Locate Field Values

Try each of the 5 methods in [`locate-field-sources.md`](locate-field-sources.md) one by one:

```
Method A: grep browser API names (covers 40%)
Method B: grep plaintext b64 keys (covers 30%)
Method C: hQ dictionary reverse-lookup (covers 50%)
Method D: algorithm magic-constant location (HMAC/UUID/...)
Method E: cross-sample diff inference (fallback)
```

If all 5 fail, **read the SDK manually** — find the `Dd()` main entry (a run of
short-named function calls) and walk the call chain one by one.

### Failure Scenario 4: Pure-Computed Generator Can't Get a Cookie

Run the failure-diagnosis decision tree in [`validate-generator.md`](validate-generator.md).
Key checks:

```
collector#1 → 200, did ob decode out state.* ?
   ↓ Yes
collector#2 → 200, does ob contain a set_cookie segment ?
   ↓ No (do:null)
   ⭐ 99% of the time it's a missing parseInt on state.no (gotchas #1)
```
