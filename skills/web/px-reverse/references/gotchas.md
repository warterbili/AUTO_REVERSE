# PX Reversing Gotcha Checklist (23 entries; #1-14 base + #15-18 strict-tier totalwine + #19-23 strict+ academy)

> Entries marked with ⭐ cost **real debugging time** — this is not a list of "things that might go wrong."

## ⭐⭐⭐ Bug #14 (discovered 2026-05-20): Replacing base64 `+` with a space when decoding the POST body

**Symptom**: When decoding a collector POST body you captured yourself (via cdp-browser / DevTools "Copy as cURL"), EV1 decodes to partial JSON and then suddenly turns to garbage.

**Root cause**: In the collector POST body PX sends, **the base64 `+` character is preserved literally** (not as `%2B`). If you process it with the form-urlencoded standard `+ → space` substitution, the base64 breaks.

```javascript
// ❌ Wrong (base64's + is eaten into a space, shifting the byte stream after b64 decode)
const payload = decodeURIComponent(raw.replace(/\+/g, '%20'));

// ✅ Right (preserve the literal +)
const payload = decodeURIComponent(raw);
```

The equivalent error on the Python side: `urllib.parse.unquote_plus(raw)` (does `+ → space`); use `urllib.parse.unquote(raw)` instead.

**General rule**: Decode a PX collector POST with **pure URI decoding** — never apply form-style processing.

## ⭐⭐⭐ Bug #1: state.no type error (the most fatal)

**Symptom**: The server returns `do:null` (PC accepted) but `ob` has only 2 segments, and `_px3` is never issued.

**Root cause**:
- The ob decoder yields `state.no` as the string `"1779131754075"`
- The SDK expects the number `1779131754075`
- Our serialize adds quotes around the string → the payload is 2 bytes longer than the SDK's
- PC validation passes (because we serialize the same way), but **the semantic validation of the actual event values** detects that the timestamp field is a string = bot

**Fix**:
```javascript
d['RTEwewNQMUg='] = parseInt(ctx.state.no);  // ⭐ always parseInt
```

**General rule**: **parseInt every numeric string coming out of the ob decoder before referencing it in an ev2 field.**

## ⭐⭐ Bug #2: UTF-8 vs Latin-1 base64

**Symptom**: The payload length is off by 30+ characters, PC does not match, and `{"do":[]}` is rejected.

**Root cause**: Under a Chinese locale, `Date.toString()` contains the CJK timezone label "China Standard Time" (rendered in Chinese characters). After XOR(50), some characters are ≥0x80. UTF-8 encodes one such character as 3 bytes, Latin-1 as 1 byte.

**Fix**:
```javascript
function b64encode(t) { return Buffer.from(t, 'utf-8').toString('base64'); }
//                                              ^^^^^^^ must be UTF-8
```

## ⭐⭐ Bug #3: anti-tamper field position destroyed

**Symptom**: Every field value is correct, but the server still does not issue `_px3`.

**Root cause**: `delete d[old_key]; d[new_key] = val` in JS appends new_key to the end of the object, **changing the field order**.

**Fix** (rebuild the dictionary to preserve order):
```javascript
var newD = {};
for (var k of Object.keys(d)) {
    if (isAntiTamperKey(k, d[k])) {
        newD[new_at_key] = new_at_val;  // occupy the original position
    } else {
        newD[k] = d[k];
    }
}
return newD;
```

## ⭐ Bug #4: anti-tamper regex range incomplete

**Symptom**: The anti-tamper key in the template is not recognized as anti-tamper.

**Root cause**: The anti-tamper key characters are state.to (0-9) ⊕ (1-12), giving a range of 0x30-0x3F = `"0123456789:;<=>?"`. The regex only covered `[0-9<=>?@]` and missed `:` and `;`.

**Fix**:
```javascript
const ANTI_TAMPER_RE = /^[0-9:;<=>?@]{15,25}$/;
```

## ⭐ Bug #5: state.appId hardcoded from the template

**Symptom**: PC is accepted but `_px3` is not issued.

**Root cause**: ob#1 supplies a different appId each time (e.g. `d85mksfkqf1c73d7p7tg` vs `d85maqv7fdnc73boge5g`). If you use the fixed value captured at template time, it will not match the current session.

**Fix**:
```javascript
d['Xi5rJBtKaB4='] = state.appId;  // from ob#1
```

**General rule**: **Every field named state.* must be decoded from the current ob — never use the template.**

## ⭐ Bug #6: SID built from the wrong UUID

**Symptom**: The `sid` parameter does not match the SDK.

**Root cause**: Used the session uuid, but the SDK uses `state.pxsid` (taken from ob#1).

**Fix**:
```javascript
sid = state.pxsid + hh(String(state.no));
```

## ⭐ Bug #7: OB decoded as utf-8

**Symptom**: The decoded ob is all garbage.

**Root cause**: The ob field generated by the SDK is binary-encoded by the server and **does not pass through** the SDK's z() function (UTF-8).

**Fix**:
```javascript
var decoded = Buffer.from(ob, 'base64').toString('binary');  // must be binary
// Wrong: .toString('utf-8')
```

## ⭐ Bug #8: Passing sid as a Python string

**Symptom**: A Python request sent with requests is rejected by the server.

**Root cause**: `sid` contains Unicode Tag Characters (U+E0100+); the bash terminal / some HTTP clients drop these invisible characters.

**Fix**:
- Check that `len(sid.encode('utf-8'))` is the expected byte count
- Python requests: escape with `urllib.parse.quote_plus` before placing it in a URL parameter
- Do not use Chrome "Copy as cURL" — it drops characters

## ⭐ Bug #9: UA mismatch

**Symptom**: The HMAC field is computed correctly but the server rejects the request.

**Root cause**: The UA in HMAC(uuid, UA) and the HTTP `User-Agent` header must be **byte-for-byte identical strings**.

**Fix**:
```javascript
const DEFAULT_UA = 'Mozilla/5.0 (Windows NT 10.0; Win64; x64) ...';

// HMAC uses this UA
d['M2MGKXUOBB8='] = hmacMD5(uuid, DEFAULT_UA);

// The HTTP header uses the same one
headers['User-Agent'] = DEFAULT_UA;
```

## ⭐⭐⭐ Bug #11: incorrect position matching of state.* → ev2 b64 key (discovered — Grubhub 2026-05-19)

**Symptom**: collector#1/#2 both return 200, all state is extracted, but `/auth/login` (or a protected endpoint) still returns 403.

**Root cause**: 4-5 field values in ev2 **come directly from ob#1's state.*** (e.g. state.no, state.to, state.appId, state.o111val). These b64 keys differ across platforms:
- iFood `state.no` is at `RTEwewNQMUg=`
- Grubhub `state.no` is at `UT0ndxdcJUQ=` (**looks like a timestamp but is actually state.no**!)

If you guess by value type ("13-digit millisecond ts → must be the pre-init timestamp"), you **miss the state.no field and fabricate the timestamp instead**. The result: when PX runs semantic validation on that field, it finds server-side state.no ≠ client value → flags as bot.

**Fix**: Do not guess by type — **match by value**. Script:
```python
# For each state.* value, find the matching b64 key in each ev2 batch
for sname, sval in state.items():
    for k, v in ev2['d'].items():
        if v == sval or (isinstance(sval, str) and sval.isdigit() and v == int(sval)):
            mapping[sname][k] += 1
# A key that hits consistently across 6+ batches → semantics confirmed
```

**General rule**: **The position of state.* (decoded from ob#1) in ev2 must be found by "value matching across 6+ batches"; guessing by type/value range is forbidden.**

## ⭐⭐ Bug #12: reusing a b64 key across event types (discovered — Grubhub 2026-05-19)

**Symptom**: Every field is correct, every type is correct, PC validation passes, but the `_px2` PX returns is still rejected at the protected endpoint. **The field count is 1 higher than the browser's.**

**Root cause**: Different event types use different b64 keys for **the same semantic**. For example:
- Grubhub `/ns duration` in **EV1** is `DFQ2Ekk4PSU=`
- Grubhub `/ns duration` in **EV2** **does not exist at all** (this field is not sent)
- iFood `/ns duration` in **EV2** is `DFg5Ekk4PSU=` (**which is also the key in its EV1**)

When I ported the iFood logic to Grubhub, I added an extra `DFQ2Ekk4PSU=` to EV2 — and one comparison was enough for PX to see "this ev type should not have this field."

**Fix**:
```python
# build_ev2 must not copy build_ev1's /ns field — check whether the template actually has it
# Use set(ev2_template.keys()) as the single authoritative field set
# DYNAMIC fields only override keys that **already exist** in the template, never add new keys
```

**General rule**: **build_evX may only override keys already present in the template; adding keys is forbidden.** A deep-clone + override pattern is far safer than building the dictionary from scratch.

## ⭐ Bug #13: PX backend IP rate limiting (discovered — Grubhub 2026-05-19)

**Symptom**: With the same generator, the first 3 runs return status=200 and obtain `_px2`/`_px3`, but after 5-10 consecutive runs it starts returning random 403s. Restarting the script lets it succeed a few more times before 403 again.

**Root cause**: The PX backend enforces a threshold on the number of collector POSTs from a single IP within a short window. Our script fires collector#1+#2 every 1.5-2 seconds, and after 10 consecutive fires gets IP-throttled. **This is not an algorithm problem; it is a behavior-pattern problem.**

**Fix**:
- When testing stability, **leave ≥10 seconds between runs**, preferably 15-30s
- In production, refresh each account through a different proxy IP (use the existing `BRIGHTDATA_RESIDENTIAL_US` rotation pool if available)
- Bootstrapping 60 accounts at once → **batch + space out**, e.g. 10 accounts per batch with a 60s gap between batches
- To confirm throttling: switch IP (turn on a hotspot / switch proxy) and retry — if it immediately succeeds, it was throttling

**General rule**: **Stability testing ≠ rapid fire.** 10/10 is a logical-correctness metric; you must add a gap between runs to avoid triggering throttling, otherwise what you are measuring is the "throttle threshold," not "algorithm stability."

## ⭐⭐⭐ Bug #14b → #15: A cookie being issued ≠ PX edge trusting it (discovered — totalwine 2026-05-25)

> ⚠️ This entry changed the **entire skill's success criteria** — read it together with [`../playbooks/validate-generator.md`](../playbooks/validate-generator.md) Layer 3.5.

**Symptom**: The full collector chain returns 200, 10/10 obtain `_px2`, and injecting into a real browser and navigating renders the page. But **using curl_cffi (chrome124 TLS, US residential proxy) to GET a PX-gated endpoint directly with that same cookie always returns 403** — the body is PX bootstrap JSON (`{"appId":...,"jsClientSrc":...}`) or px-captcha challenge HTML. An empty cookie and our cookie **return the exact same 403 body** at this endpoint, as if PX never saw our cookie.

**Root cause**: PX has **two validation layers**:

```
Layer 1 — SDK client-side validation (cookie signature h, structure, TTL)
  → Our cookie passes this layer, so after injection into the browser the SDK does not re-fire the collector.

Layer 2 — PX backend trust-score validation
  → During the collector chain, the PX backend runs semantic analysis on EV1/EV2/EV3 fields and scores them.
  → Low score → the cookie is still issued (this is a "lenient signal" that makes the bot think all is well),
    but the backend session record is marked trust=low → the edge treats all subsequent
    requests carrying that cookie as "unauthorized" → 403 PX block.
```

**Key evidence (apples-to-apples control)**:
- The browser's own `_px2` + any BrightData US proxy IP + curl_cffi → ✅ 200
- Our generated `_px2` + the same proxy IP + the same curl_cffi → 🚫 403 PX-block
- ⇒ Transport is fine; **the problem is the cookie content** — not a cryptographic error, but a low trust score.

**Fix (diagnostic path)**:

1. **Step 1**: Run `compare_ev2_field_by_field` ([bundled with the skill](../scripts/compare_ev2_field_by_field.py)) — our EV2 vs the real captured EV2 field set + types + STATIC values. 99% of root causes surface here.
2. **Step 2**: Run [`recover-hmac-formulas.md`](../playbooks/recover-hmac-formulas.md) to verify the input of each HMAC/MD5 field is correct.
3. **Step 3**: Check [`deployment-tiers.md`](deployment-tiers.md) to see whether this site is a strict-tier deployment → if so, also run the Gotcha #16/#17 checks.

**Do not**:
- ❌ Suspect the TLS fingerprint (chrome124 impersonation is sufficient)
- ❌ Suspect that the IP is flagged (switching proxies does not help — proof: the browser cookie returns 200 across any proxy IP)
- ❌ Rewrite the collector POST timing

**General rule**: **Obtaining the cookie is not a win. Validation Layer 3.5 (= the cookie actually hits a PX-gated endpoint and returns real content) is the win.** iFood/Grub's `business_api_demo.js` is the template; any new site's README must have an equivalent.

## ⭐⭐⭐ Bug #16: Seq=2 is not a "cleanup ping" — in a strict-tier deployment it is a mandatory beacon (discovered — totalwine 2026-05-25)

**Symptom**: The collector chain runs 2 POSTs (seq=0, seq=1), obtains the cookie, and stops. The generator test passes 10/10. But when that cookie is run through the real-endpoint test from Gotcha #15, it is rejected.

**Root cause**: In a strict-tier deployment the SDK sends a **3rd collector POST (seq=2, en=NTA)**. This POST body contains the `OkpJAH8oTTA=` field, whose value is **the `_px2` cookie string just obtained**.

```
seq=0 (EV1)  → PX returns state (no/to/qa/vid/pxsid/cts/appId/hid…)
seq=1 (EV2)  → PX issues the _px2 cookie (delivered but trust=pending)
seq=2 (EV3)  → the client echoes _px2 back to PX → PX marks trust=verified
```

Skipping seq=2 → the cookie stays at `trust=pending` forever → the edge does not admit it when seen (behaves like unauthorized).

**How to recognize this trap**:
- Inspect `collector_post_count` in `meta.json` — if it is ≥ 3 and you see `request_3.txt` exists
- Decode the payload of the `request_3.txt` body and check whether it contains `OkpJAH8oTTA=` (value is the base64 cookie)
- If so → you must implement the EV3 + seq=2 POST

**Fix**:
```js
// After obtaining the cookie, append:
const ev3 = buildEv3({ uuid, sendTime, cookieValue });   // cookieValue = the _px2 just obtained
const payload3 = generatePayload(ev3, state.no, uuid);
const pc3 = generatePC(ev3, uuid, TAG, FT);
const body3 = formEncode({
    payload: payload3, appId, tag, uuid, ft, seq: 2, en: 'NTA', bi,
    cs: state.qa, pc: pc3, sid, vid: state.vid, cts: state.cts,
    hid: state.hid, rsc: 3,
});
await post(COLLECTOR_URL + '?seq=2&rsc=3', body3, ua);
```

**General rule**: **The number of collector POSTs in the capture = the number you must send.** Do not assume everything after the 3rd is "beacon noise" — decode it first to check whether it is a cookie echo.

## ⭐⭐ Bug #17: Counter sub-field synchronization constraint (discovered — totalwine 2026-05-25)

**Symptom**: Every b64 key in EV2/EV3 is correct, the field types and counts all match, but PX still scores the session low.

**Root cause**: A strict-tier deployment validates **the correlation between sub-fields of a dictionary-type field**. The totalwine case:

```js
'MDxDNnVeQgQ=': {
    PX12738: <num>,
    PX12739: <num>,
    PX12740: 0,
    PX12741: -1,
}
```

Across 6 real batches:
- **PX12738 always equals PX12739** (or both are 0)
- It is **monotonically increasing** across EV1 → EV2 → EV3 (EV1 is typically 1, EV2 is 1000–3000, EV3 is 2000–9000)

Our generator previously assigned two independent `Math.random()` values (e.g. EV2 gave 520 / 2072) → in the real data **6/6 batches are equal** → the PX backend sees "the same counter reported inconsistently on two channels" across events → flags as bot.

**Fix**:
```js
const n2 = 1000 + Math.floor(Math.random() * 2500);
ctx._counter2 = n2;
tpl['MDxDNnVeQgQ='] = { PX12738: n2, PX12739: n2, PX12740: 0, PX12741: -1 };

// EV3:
const n3 = (ctx._prevCounter || 1500) + 1500 + Math.floor(Math.random() * 5000);
tpl['MDxDNnVeQgQ='] = { PX12738: n3, PX12739: n3, PX12740: 0, PX12741: -1 };
```

**General rule**: **For any dictionary-type DYNAMIC field**, the diff tool must check not only whether the field is present and whether the value is STATIC, but also **whether there are correlation constraints between sub-fields** (equality, sum to 0, monotonic increase, etc.). This applies to **all deployments** — iFood/Grub simply got lucky that their random values never crossed the relevant thresholds.

## ⭐⭐ Bug #18: HMAC field inputs must be **measured live**, never copied across sites (discovered — totalwine 2026-05-25)

**Symptom**: Reusing iFood's or Grubhub's HMAC formulas (e.g. `hmac(uuid, UA)`, `hmac(uuid+':a', UA)`, `hmac(uuid+':b', UA)`) on a new site produces HMAC field values that are **structurally correct** (32-char hex), **type-correct** (string), and **length-correct**, and PC validation passes — but the cookie is still rejected afterwards.

**Root cause**: The 4 HMAC/MD5 fields (e.g. `Cho5UEx3PWY=`, `Lx8cFWl9HCE=`, `UiJhKBREYhs=`, `EFwjFlU8JyU=`) have **identical b64 keys across different sites' SDKs** (same obfuscation dictionary), but **different inputs**:

| Field | iFood/Grub (old doc's guess) | **totalwine measured** |
|---|---|---|
| `Cho5UEx3PWY=` | `hmac(uuid, UA)` | `hmac(uuid, UA)` ✓ happens to match |
| `Lx8cFWl9HCE=` | `hmac(uuid+':a', UA)` ❌ | `hmac(state.vid, UA)` |
| `UiJhKBREYhs=` | `hmac(uuid+':b', UA)` ❌ | `hmac(state.pxsid, UA)` |
| `EFwjFlU8JyU=` | `hmac(uuid+':c', UA)` ❌ | **`md5(state.vid)` (single arg, not HMAC!)** |

The PX backend recomputes each field per site using the SDK's **real** input and compares against what the client reported. One wrong HMAC → bot score.

**Fix (SOP)**: See [`../playbooks/recover-hmac-formulas.md`](../playbooks/recover-hmac-formulas.md). Short version:

1. grep the SDK for the b64 key's occurrence → find the `i["..."] = jm(X(), Y())` pattern
2. Find the X(), Y() function definitions (`jm` = HMAC-MD5; single arg = MD5)
3. Enumerate candidate inputs: uuid / state.vid / state.pxsid / state.cts / state.qa / sessionStorage[?]
4. Crypto-verify across 6 batches: whichever input formula makes 6/6 batches match → formula confirmed
5. Do not stop at the first one — all 4 HMAC fields must be verified individually

**General rule**: **Any HMAC/MD5 field must be re-verified across 6 batches via crypto on a new site. Never copy assumptions from iFood / Grub.** This applies to **all future sites** — it is not specific to totalwine.

## ⭐⭐⭐ Bug #19: A single-snapshot diff cannot see "cross-event consistency" — the hidden root cause of low trust on strict tiers (discovered — academy 2026-06-12)

**Symptom**: The generator passes every existing diff (`diff_ev_ours_vs_real` / `compare_ev2_field_by_field` /
`diff_samples`: field set ✅, types ✅, STATIC values ✅, shapes ✅, counter single-event invariant ✅),
the collector chain is all 200, the cookie is issued normally — but when used against a PX-gated endpoint it is **consistently rejected with low trust**.
You keep re-checking individual EV2 fields; each one "looks correct," and you just cannot find the error.

**Root cause**: The existing diff tools **structurally compare only a single event snapshot**. But the strict-tier backend validates relationships that exist only **between the EV1→EV2→EV3 sequence**, and these constraints are invisible in any single snapshot:

```
CONSTANT  fields: must have exactly the same value in every event of a session
          (page-load timestamp, uuid, navigator.platform, location.href)
MONOTONIC fields: must strictly increase across events
          (performance.now timer, event counter, current timestamp)
DICT/counter sub-field cross-event invariants
          (PX12738==PX12739, a sub-counter that is constantly 0 or mirrors another, and grows monotonically over time)
```

**The real trap**: academy's generator gave `ZRBUGyN3Wiw=` (the page-load timestamp) a **different** value in
EV1/EV2/EV3 each time — a single EV2 looks perfectly normal (just a reasonable timestamp), but it should have been
**constant** across the three events. The backend sees EV1≠EV2 and immediately flags as bot. Likewise `T34+NQkfPgc=`
(a perf timer): I faked it with a random value of the right magnitude (fooling the shape diff), but it did not increase
monotonically across events, nor track the counter → low trust.

**Key realization**: these rules **are already present in real CDP captures** — every capture containing seq=0/1/2 is
the EV1/EV2/EV3 of one session. No node_bridge needed; **running cross-event analysis on real packets extracts the rules.**
They were missed before only because there was no tool doing the "cross-event" dimension of comparison.

**Fix (mandatory steps, add to the diff flow)**:

```bash
# 1. First extract rules from real captures (which fields are CONSTANT / which are MONOTONIC^)
node scripts/decode_payload.js samples/1/request_1.txt > samples/1/decoded_payload_1.json  # x3 events x N batches
python scripts/cross_event_consistency.py samples            # real packets → rule table

# 2. Have the generator dump its own EV1/EV2/EV3, run the same tool
python scripts/cross_event_consistency.py samples/_our_ev    # our output

# 3. Compare line by line: for every CONSTANT / MONOTONIC^ field flagged "<-- generator MUST preserve",
#    the generator's verdict must match the real packets. A mismatch = a cross-event bug = low trust.
```

Implement in the generator:
- CONSTANT fields (page-load timestamp, etc.): **compute once, reuse the same value across the three events** (do not recompute per event).
- MONOTONIC fields (perf/counter/now-ts): derive them from **one virtual clock** `perf = sendTime - initTime`,
  guaranteeing EV1<EV2<EV3, with the counter coupled to perf.

**General rule**: **"field set + types + static values + shapes" all matching ≠ correct. Strict sites must additionally run
`cross_event_consistency.py` so that the cross-event behavior of every CONSTANT / MONOTONIC field also matches the real packets.**
This dimension is universal to all strict PX sites.

## ⭐⭐⭐ Bug #20: Counter sub-field "legal pattern space" — `(0,N)` is an illegal combination (academy 2026-06-13, the precise version of #17)

**Symptom**: The counter field (academy `AEwwBkUuMjQ=`) is fully correct on field set / types / monotonicity, the cookie is issued,
but the PX-gated endpoint **steadily oscillates around a ~40% pass rate**. Change it → straight to 10/10. This is the **real reason** academy was stuck.

**Root cause**: For the counter dict `{PX12738:N, PX12739:x, PX12740:y, PX12741:-1}`, the two sub-fields
`PX12739/PX12740` can **only be 0 or ==PX12738, never independent**. Cross-tabulating 6 real captured batches, a real browser
**produces only three patterns**: `(0,0)`, `(N,N)`, `(N,0)`. **`(PX12739=0, PX12740=N)` never appears in the real data** →
the strict tier flags it as bot instantly. The earlier #17 recorded only `PX12738==PX12739` and **missed the PX12740 dimension**,
so academy was wrongly given `(0,N)`.

```javascript
// ❌ Wrong (an illegal combination that does not exist in the real data)
counter = n => ({ PX12738: n, PX12739: 0, PX12740: n, PX12741: -1 });
// ✅ Right (use the real pattern that matches the captured batch; academy measured: batch1,2=(0,0) batch3=(N,N) batch4-6=(N,0))
const CTR_PAT = { 1:[0,0], 2:[0,0], 3:[1,1], 4:[1,0], 5:[1,0], 6:[1,0] }[pick] || [1,0];
counter = n => ({ PX12738:n, PX12739: CTR_PAT[0]?n:0, PX12740: CTR_PAT[1]?n:0, PX12741:-1 });
```

**General rule**: For counter-class dict fields, **cross-tabulate every sub-counter across 6 batches**, listing all the
`(sub-fields...)` combinations that ever appeared = the legal pattern space; the generator may only emit one of them.
Don't only verify "two are equal" — verify "the combination is legal."

## ⭐⭐⭐ Bug #21: The `/ns` token is **TLS-fingerprinted** — one fetched by node leaks the identity (academy 2026-06-13)

**Symptom**: The `/ns` token field of EV2/EV3 (academy `MV0BV3Q9AGE=`, i.e. `sm`) is **shorter** than the real capture
(432 vs the real 504-512); a field-by-field diff spots the length mismatch at a glance; the cookie is low trust.

**Root cause**: The `/ns?c=<uuid>` endpoint encodes the **client TLS fingerprint** into the returned token. node `https`'s JA3 → 432 bytes,
real Chrome (curl_cffi `chrome142`/`chrome131`) → 504-512. Fetching /ns with node stamps the token with
"claims Chrome but is actually node" → the backend downgrades trust.

```javascript
// ❌ Wrong: fetch /ns with node https (the token carries node's TLS fingerprint)
const ns = await fetchNs(uuid, NS_HOST);          // sm.length == 432
// ✅ Right: /ns and the collector POST ride the same Chrome-impersonate session (curl_cffi chrome142)
//    → sm.length == 504-512, matching the real capture
```

**General rule**: On strict sites, **`/ns` must ride the same real Chrome TLS transport as the collector**. Verification method:
fetch `https://<ns-host>/ns?c=<uuid>` once with node and once with curl_cffi chrome, and compare the `sm` length.

## ⭐⭐ Bug #22: A pure-math static template is most reliable when **captured from real Chrome CDP** — but JSDOM is **not** a trust ceiling (academy 2026-06-13; corrected 2026-06-13)

**⚠️ Correction first**: this entry was originally written as "JSDOM/node_bridge can't pass, fit only as an oracle" — **that was wrong**,
misled by a burned exit IP (the real cause is Bug #23). **node_bridge running the live SDK (JSDOM live) passes academy strict+ in
practice on a clean residential IP, pulling 1.2MB of real product data.** The JSDOM fingerprint is **not** a hard trust ceiling.

**What still holds**: when writing **pure math**, the static EV template should **preferably be captured from real Chrome CDP**
(`decoded_payload_2.json`, academy's real capture has **203 fields**, more complete than the 177 a JSDOM dump gives, with truer
sensor values) — this is the **safe default**, *not* because a JSDOM static template is necessarily rejected (no A/B has been done
on the same clean IP between a JSDOM static template and a real-Chrome template, so it cannot be asserted). **Rotating multiple real
fingerprints** per mint, to avoid "minting many cookies from the same fingerprint" being correlated and downgraded, is still good practice.

**The real determinant of academy's pass rate is Bug #23** (legal counter + chrome142 TLS transport + clean mint IP),
**not** the JSDOM-vs-real-Chrome template debate. Back then "swapping the template made it pass" was very likely the confounded
result of simultaneously switching to a clean IP.

**General rule (corrected)**: node_bridge has **two** legitimate uses — (1) **a reversing oracle** (see how the live SDK computes a field);
(2) **a maintenance-free production fallback** — it runs the live SDK, auto-adapts when the SDK rotates, and on a clean IP can directly
produce a gateway-passing cookie. Pure math (which needs a re-capture + template rebuild to follow an SDK upgrade) and node_bridge
(auto-adapting) are **two complementary production lines**.

## ⭐⭐⭐ Bug #23: strict+ trust is bound to "transport TLS + IP mint reputation" (academy 2026-06-13)

**Symptom**: Every EV field is correct field-by-field (diff passes), the counter is legal, /ns is 504, **the cookie content is impeccable on its own**,
yet the pass rate still jitters (same code: fresh residential IP 10/10, the local IP after a re-test 1/5, the proxy pool after heavy hitting drops to 0).

**Root cause**: academy binds trust additionally to the **transport authenticity at mint time + the exit IP reputation**:
1. **Transport**: must be a curl_cffi `chrome142` persistent session (/ns+collector+edge over one TLS); minting over node TLS → low trust.
2. **IP mint reputation**: a single exit IP minting at high frequency gets rolling-downgraded by PX per IP (or even the whole proxy pool).

**Localization method (4-way matrix)**: hit the gateway with the four groups of real browser/curl × real cookie/our cookie —
- REAL cookie passes both + OUR cookie fails both → cookie content problem (keep diffing)
- OUR cookie **passes in real browser, fails under curl** → the cookie is borderline trust (like academy before the counter fix)
- REAL cookie also starts failing under curl → **the IP/proxy pool was polluted by your testing**; switch IP and re-test, don't misread it as an algorithm regression

**General rule**: On strict+ sites, **Layer 3.5 must be tested on a clean residential IP**, one new IP per cookie;
"the browser can open the homepage" ≠ "automated minting from that IP is trusted" (the homepage is usually not strictly gated).
When the pass rate is low and jittery, **suspect IP/transport first**; falsify it with the 4-way matrix before going back to check fields.
